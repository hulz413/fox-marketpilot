## 背景

MarketPilot 目前已经完成项目骨架，`frontend/` 具备 Next.js、TypeScript、Tailwind CSS、shadcn/ui 和 feature 占位结构，但首页仍是“项目骨架已就绪”的技术占位页。下一步需要把前端从工程骨架推进到产品骨架，让用户首次打开时能理解这是一个商机顾问 Agent，并能看到 P0 演示闭环的主要入口。

本 change 采用已确认的浅色企业列表型视觉方向：左侧边栏、顶部页面栏、中间主内容区，整体像克制的企业后台，但信息架构保持单用户 MVP 演示优先，不扩展成复杂经营看板。产品默认主色为 `#2F6F4E` 深翡翠绿。

## 目标 / 非目标

**目标：**

- 用企业列表型应用壳替换现有首页占位，让 MarketPilot 的产品定位、主要入口和页面结构可浏览。
- 使用 `#2F6F4E` 深翡翠绿作为产品主色，传达增长、机会和可信顾问感。
- 只保留 P0 必要入口：新建研究、研究任务、商机推荐、最终报告、研究历史。
- 为后续 P0 changes 预留稳定页面路由和 feature 边界。
- 用中文演示数据展示首次空状态、示例研究、运行中、失败和完成等关键状态。
- 保持浅色、克制、可扫描的企业后台风格，避免暗色主题和营销式落地页。

**非目标：**

- 不实现真实研究任务创建、Agent 执行、SSE 进度订阅或后端数据读写。
- 不实现登录、团队协作、权限、支付、自动下单、自动联系供应商或自动发布内容。
- 不把来源依据、供给候选、竞品情报、风险复核、验证计划做成左侧一级入口；这些能力后续作为商机详情或报告内部分区扩展。
- 不新增第三方 UI 或状态管理依赖。

## 技术决策

### 决策：使用企业列表型应用壳作为默认产品形态

采用左侧边栏 + 顶部页面栏 + 中间内容区的浅色企业应用壳。左侧承载 P0 导航，顶部承载当前页面标题和主操作，中间根据路由展示任务列表、推荐列表、详情、报告或历史骨架。

备选方案：

- 多 Tab 风格候选页：适合设计探索，但不适合进入真实产品骨架。
- 营销式首页：能解释产品价值，但会推迟用户进入研究流程，不适合 MVP 演示闭环。
- 复杂三栏工作台：信息密度高，但会过早暴露 P1/P2 分析模块，增加首次理解成本。

### 决策：使用 `#2F6F4E` 深翡翠绿作为产品主色

主按钮、品牌图标、左侧选中态、推荐状态和 Agent 运行中的关键状态使用 `#2F6F4E`。背景使用轻暖灰绿，边框使用中性浅灰绿，机会或风险提示可使用低饱和琥珀色作为辅助强调。

备选方案：

- 森林绿：更传统企业后台，但更容易靠近采购或管理系统气质。
- 青黛蓝绿：更 AI 和数据感，但增长、机会的业务联想弱一些。
- 沉静靛蓝：更通用 SaaS，可信但辨识度较弱。

### 决策：左侧导航只保留五个 P0 入口

左侧导航固定为：新建研究、研究任务、商机推荐、最终报告、研究历史。来源、供给、竞品、风险和验证计划不作为一级入口，而是在商机详情、最终报告或后续 P1 页面内呈现。

备选方案：

- 把所有 roadmap 模块都放进左侧：看起来完整，但会让产品像复杂后台，并让尚未实现的 P1 能力抢走 P0 主线注意力。
- 只保留一个“工作台”：入口最少，但后续 P0 页面难以被用户和测试明确访问。

### 决策：页面骨架使用演示数据和占位状态，不接真实 API

本 change 的目标是产品骨架可浏览，因此页面可以使用本地静态演示数据和组件状态展示 P0 所需的视觉结构。真实 API、SSE 和持久化由后续 `create-research-task`、`show-research-progress`、`run-opportunity-research` 等 changes 接入。

备选方案：

- 直接串后端占位接口：会扩大范围，把产品骨架和任务闭环混在一起。
- 完全空页面：实现简单，但无法验证企业列表型风格和关键状态。

### 决策：路由边界先围绕 P0 闭环命名

建议产品骨架预留这些前端路由：`/` 或 `/research/new`、`/research/tasks`、`/opportunities`、`/opportunities/[id]`、`/reports/[id]`、`/history`。导航和按钮应指向这些骨架页面，后续 change 可在同一路由内补真实行为。

备选方案：

- 使用单页状态切换：原型速度快，但不利于后续按垂直切片实现和验收路由可访问性。
- 细分大量 P1 路由：会提高导航复杂度，并超出 P0 产品骨架范围。

## 风险 / 取舍

- [Risk] 静态演示数据可能让用户误以为研究逻辑已完成 → Mitigation: 页面文案使用“示例”“占位”“后续接入”等产品化说明，并避免展示真实运行承诺。
- [Risk] 企业后台风格容易变成复杂管理系统 → Mitigation: 限制一级导航数量，P1 分析模块只作为详情/报告内部区块预留。
- [Risk] 产品骨架路由后续可能和真实 API 数据模型不完全匹配 → Mitigation: 路由按用户流程命名，不按临时数据结构命名，降低后续迁移成本。
- [Risk] 中文界面和英文代码命名需要同时维护 → Mitigation: 用户可见文案保持中文，结构化路径、组件和类型使用英文。
