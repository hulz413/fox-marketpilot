## 背景

MarketPilot 现在已经完成基础商机推荐、公开来源收集、来源透明度、需求洞察、货源候选和竞品参考。用户可以看到“为什么可能有需求”“可以从哪里继续找货”和“市场上有哪些类似参考”，但还缺少“首批验证大概要花多少钱、售价和毛利是否粗略成立、最需要验证的预算假设是什么”的结构化判断。

`estimate-validation-budget` 是竞品参考之后的 P1 质量增强切片。它复用当前研究任务、商机、货源候选、竞品参考和 Agent 阶段事件能力，为每个商机生成可读、可追踪、仍需验证的预算粗算，不改变 P0 基础商机生成边界。

## 目标 / 非目标

**目标：**

- 为每个商机生成并保存 1 条结构化验证预算估算。
- 覆盖预估拿货成本、预估售价、粗略毛利空间、首批验证数量、首批验证预算和关键验证假设。
- 优先使用商机价格带、粗略利润空间、任务预算、期望利润、货源候选和竞品参考，缺少上下游数据时仍生成谨慎的待验证粗算。
- 在商机详情展示完整验证预算，在基础报告展示摘要。
- 为验证预算估算阶段记录 Agent 运行事件、日志和 LangSmith trace 元数据。
- 本地和测试环境提供确定性回退，自动化测试不依赖外部 LLM、搜索或网络服务。

**非目标：**

- 不做精确财务模型、现金流模型、税费模型或多平台手续费模型。
- 不确认真实采购价、真实成交价、库存、运费、退货率或转化率。
- 不做风险复核或完整行动计划；这些由后续 P1 切片处理。
- 不让验证预算反向修改商机排序、风险等级、价格带或基础推荐理由。
- 不引入 RAG、向量检索、LangSmith 数据集或自动评测；这些留给 `evaluate-rag-quality`。
- 不把预算估算表达为“利润已确认”“保证回本”“确定毛利”或等价结论。

## 技术决策

### 1. 新增 `validation_budgets` 独立模型

验证预算作为商机结果的 P1 增强数据保存，不直接扩展 `opportunities` 表。建议新增 `validation_budgets` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `research_task_id`: 内部外键，关联研究任务。
- `opportunity_id`: 内部外键，关联商机。
- `estimated_unit_cost`: 预估单件拿货成本或成本区间，使用字符串保留粗略区间和币种。
- `estimated_selling_price`: 预估售价或售价区间，使用字符串保留粗略区间和币种。
- `rough_gross_margin`: 粗略毛利空间或毛利率表达。
- `first_batch_quantity`: 建议首批验证数量，允许区间表达。
- `first_batch_budget`: 首批验证预算粗算，允许区间表达。
- `key_assumptions`: JSON 字符串列表，记录最需要验证的关键假设。
- `calculation_note`: 预算估算说明，解释主要依据和不确定性。
- `estimate_status`: `derived`、`fallback` 或 `insufficient_data`，表达估算依据状态。
- `created_at`、`updated_at`、`deleted_at`。

该表遵循项目数据库通用字段规范；默认读取只返回 `deleted_at` 为空的记录。重新运行研究任务并成功生成新商机后，旧商机关联的验证预算应被软删除或从默认读取中排除。

替代方案：把验证预算塞进 `opportunities` 的 JSON 字段。这样能少建表，但会让 P0 基础推荐和 P1 分析结果耦合，也不利于后续风险分析和行动计划复用预算假设。

### 2. 不新增来源关联表

验证预算第一版是对已有结构化分析的派生粗算，主要依据来自商机字段、货源候选和竞品参考，而不是直接引用新的外部来源。因此第一版只保存 `estimate_status` 和 `calculation_note`，不新增 `validation_budget_sources`。

如果后续要把每条预算假设绑定到具体来源，可以在单独 change 中增加关联表。当前保持模型简单，避免为了粗算过早引入复杂证据链。

### 3. 验证预算生成放在竞品参考之后

LangGraph 节点顺序调整为：

```text
normalize_intake
  -> generate_opportunities
  -> validate_results
  -> persist_results
  -> collect_research_sources
  -> generate_demand_insights
  -> generate_supply_candidates
  -> generate_competitor_references
  -> estimate_validation_budgets
  -> END
```

生成节点读取当前任务的未软删除商机、可用需求洞察、可用货源候选和可用竞品参考。预算估算优先使用货源候选里的价格区间和起订量、竞品参考里的售价区间、商机自带价格带和粗略利润空间；缺少任一输入时仍可基于任务预算和商机字段生成谨慎粗算，并用 `fallback` 或 `insufficient_data` 标识。

替代方案：在货源候选之后、竞品参考之前生成预算。这样能更早产出预算，但会少用竞品售价参考，预算判断质量较弱。

### 4. 使用单次批量结构化 LLM 调用

单个任务最多 5 个商机，验证预算适合用一次批量 LLM 调用生成。输入包括任务上下文、商机基础字段、裁剪后的需求摘要、货源候选摘要和竞品参考摘要，输出为按商机 UUID 映射的 JSON。后端使用 Pydantic 校验字段、数量、排序、文本长度和谨慎表达；校验失败可重试一次，最终失败时记录阶段失败。

本地和测试环境缺少 LLM key 时，使用确定性回退基于商机字段、货源候选和竞品参考生成稳定中文预算粗算。生产环境缺少 LLM key 时按配置错误处理，但验证预算失败不影响基础结果。

替代方案：每个商机单独调用 LLM。隔离性更好，但成本和延迟更高，且 MVP 每次只有 3-5 个商机，批量更合适。

### 5. 验证预算失败不破坏基础结果

验证预算是 P1 增强能力。若商机生成、校验或持久化失败，任务仍按现有核心逻辑失败；若验证预算生成失败，任务保留 `completed`，已生成商机、来源、需求洞察、货源候选和竞品参考仍可读取，`estimate_validation_budgets` 阶段事件记录为失败，验证预算 API 返回空结果或空状态。

这样能避免 LLM 波动或上游数据不足破坏已完成的 P0 演示闭环，同时让演示和排障能通过阶段事件看到预算估算缺失原因。

替代方案：验证预算失败即整个研究任务失败。这样能保证完成任务都有预算估算，但会让 P1 增强能力影响核心闭环稳定性。

### 6. 前端只在结果页面内部展示

验证预算不新增一级导航。商机详情展示完整预算估算，包括预估拿货成本、预估售价、粗略毛利空间、首批验证数量、首批验证预算、关键假设和估算说明。基础报告展示每个商机的预算摘要和进入详情的入口。缺少验证预算或读取失败时，页面保留基础商机内容，并展示中文非阻塞空状态。

文案继续使用“粗略估算”“待验证假设”“首批验证预算”“需要确认”等谨慎表达，不使用“利润已确认”“保证回本”“确定毛利”等结论式表达。

## 迁移计划

1. 新增数据库迁移，创建 `validation_budgets`，包含 UUID、时间戳和软删除字段。
2. 新增后端模型、仓储、服务和 API，先保证空结果和 fallback 可用。
3. 将 `estimate_validation_budgets` 接入研究任务阶段枚举、LangGraph 节点和前端阶段文案。
4. 增加前端读取函数和展示组件，接入商机详情与基础报告。
5. 补齐后端、前端和 OpenSpec 校验后再进入实现完成状态。

回滚时可以停止调用新节点并隐藏前端面板；新表作为 P1 增强数据保留，不影响已有基础商机、来源、需求洞察、货源候选或竞品参考读取。

## 风险 / 取舍

- [风险] 用户误以为预算估算是精确财务测算 -> 缓解：规格、生成提示和 UI 均要求谨慎表达，并展示关键假设。
- [风险] 货源或竞品数据不足导致预算泛化 -> 缓解：允许 `fallback` 和 `insufficient_data` 状态，明确预算是待验证粗算。
- [风险] 新增 LLM 节点增加耗时 -> 缓解：单次批量生成，限制输入摘要数量和文本长度。
- [风险] 验证预算失败影响演示节奏 -> 缓解：非阻塞失败，保留基础结果并记录阶段事件。
- [风险] 重新运行后旧预算污染新结果 -> 缓解：按任务或商机软删除旧预算，默认读取只返回当前未软删除数据。

## 开放问题

- 第一版是否需要把 `first_batch_budget` 拆成拿货、样品、运费和内容投放四个子项。建议第一版保持单字段粗算，避免提前进入精确财务模型。
