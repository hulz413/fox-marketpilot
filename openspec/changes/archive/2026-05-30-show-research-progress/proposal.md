## 为什么

当前 P0 已经跑通“创建任务 -> 启动基础研究 -> 生成商机 -> 查看列表/详情/基础报告”的闭环，但用户提交任务后只能在任务列表看到粗粒度状态和当前阶段，仍然像在等待黑盒结果。`observe-agent-runs` 已经持久化阶段事件和耗时，本 change 将这些运行数据转化为用户侧可理解的进度页，补齐 P0 演示中的等待、失败和重新运行体验。

## 变更内容

- 新增研究任务进度能力：用户可以打开单个任务进度页，查看任务输入摘要、状态、当前阶段、已完成阶段、阶段耗时、失败原因、LangSmith trace 入口和下一步操作。
- 新增阶段事件读取契约：前端可以按任务 UUID 和当前 run ID 读取最新运行的阶段事件时间线，不暴露内部自增 ID。
- 调整任务列表操作入口：运行中、失败、已创建任务可以进入进度页；已完成任务仍可进入商机推荐和基础报告。
- 失败任务在进度页展示中文失败原因，并提供重新运行入口；重新运行后进度页展示新 run 的状态和阶段。
- 完成任务在进度页提供进入商机推荐和基础报告的入口。
- 不引入 SSE、WebSocket、外部调研、来源收集、RAG、竞品深挖或多 Agent 协作。

## 能力

### 新增能力

- `research-progress`: 定义研究任务进度页、阶段时间线、失败展示和完成后跳转的用户侧进度查看能力。

### 修改能力

- `research-tasks`: 任务读取和任务列表需要提供稳定的进度页入口，并支持用户从任务进度页重新运行失败或已创建任务。
- `agent-run-observability`: 已持久化的阶段事件需要通过用户侧可读取接口返回给任务进度页，并保持不暴露内部自增 ID 或敏感错误信息。

## 影响

- 后端 API：新增或扩展研究任务进度读取接口，返回任务摘要、当前运行关联和阶段事件。
- 后端 schemas/service/repository：补充阶段事件对外读取模型和按任务最新 run 查询逻辑。
- 前端页面：新增任务进度页路由和进度组件；任务列表操作入口调整为进入进度页。
- 前端数据请求：新增进度读取 API 和运行状态轮询。
- 测试：覆盖进度接口、阶段事件读取、失败展示、重新运行入口和完成后跳转入口。
