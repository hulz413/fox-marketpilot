## 背景

`create-research-task`、`run-opportunity-research` 和 `observe-agent-runs` 已经提供了任务生命周期、基础商机生成、运行 ID、trace URL、当前阶段、失败原因以及 `agent_run_events` 阶段事件落库。当前前端主要在研究任务列表中展示任务状态和当前阶段，用户提交后需要回到列表等待，无法集中理解“现在执行到哪里、哪些阶段完成、失败在哪里、下一步能做什么”。

本 change 将已有任务状态和阶段事件组合成研究进度体验。它不改变基础研究 Agent 的执行边界，不新增外部调研或来源能力，也不引入新的数据库表。现有表继续遵循项目通用字段规范：内部自增 `id` 仅用于数据库关联，对外使用任务 UUID、事件 UUID、run ID 和 trace ID；读取软删除数据时默认只返回 `deleted_at` 为空的任务与事件。

## 目标 / 非目标

**目标：**

- 为单个研究任务提供进度页，展示任务摘要、状态、当前阶段、阶段时间线、耗时、失败原因和相关操作。
- 为前端提供稳定的进度读取模型，组合研究任务和当前 run 的阶段事件。
- 让任务列表、新建研究成功页、失败重新运行流程都能自然进入进度页。
- 在运行中使用轻量轮询刷新状态，完成或失败后停止轮询并提供下一步入口。
- 保持用户可见失败信息安全，不暴露内部自增 ID、原始异常堆栈或敏感凭证。

**非目标：**

- 不实现 SSE、WebSocket 或实时 token 流。
- 不新增 Agent 节点，不改变基础商机生成逻辑。
- 不收集公开来源、不做 RAG、不做竞品或供给深挖。
- 不新增数据库表；如现有阶段事件字段足够，本 change 仅补充读取模型和 API。
- 不实现完整历史详情对比；进度页默认展示当前 run，旧 run 历史保留在数据层。

## 技术决策

1. **使用组合式进度读取接口，而不是让前端拼多个接口**

   新增 `GET /api/v1/research-tasks/{task_uuid}/progress`，返回任务公开字段、当前 run 关联字段、当前阶段、失败原因、trace URL、阶段事件列表和可用操作提示。前端仍可复用现有任务和商机接口，但进度页不需要先读任务再读事件。

   备选方案是新增独立 `GET /api/v1/research-tasks/{task_uuid}/runs/{run_id}/events`。该方案更底层，但前端需要处理 run ID 为空、旧 run、新 run、任务软删除和失败信息组合，用户侧进度页会分散更多业务判断。因此本 change 采用组合式接口，并在 service 层复用 agent run repository。

2. **默认展示当前 run 的事件时间线**

   进度页读取任务当前 `run_id` 对应的未软删除事件，按 `started_at` 升序展示。重新运行后任务记录会刷新为新 run ID，进度页自动展示新 run 的阶段事件；旧 run 事件不删除、不覆盖，但不作为默认进度展示。

   备选方案是展示全部运行历史。该方案对排障有价值，但会扩大 P0 页面复杂度，且与 `browse-research-history` 或后续运行详情能力重叠。

3. **阶段展示由固定阶段顺序和事件状态共同驱动**

   前端使用已存在的阶段枚举映射中文文案，并按 `queued -> normalize_intake -> generate_opportunities -> validate_results -> persist_results -> completed` 展示主要进度。已有事件决定阶段是否已开始、完成或失败；任务 `current_stage` 用于处理尚未写入事件的 `queued` 或失败状态。

   备选方案是在后端计算百分比并返回。该方案初期更省前端逻辑，但会把展示进度和业务状态耦合得过早。当前阶段数量固定，先由前端派生更灵活。

4. **运行中使用 TanStack Query 轮询，不引入 SSE**

   进度页在任务状态为 `queued` 或 `running` 时每 2-5 秒刷新一次进度读取接口；进入 `completed` 或 `failed` 后停止轮询。技术栈已经预留 SSE，但当前 P0 基础研究只有少量阶段，轮询足以支撑演示。

   备选方案是立即实现 SSE。该方案更实时，但会增加后端连接管理、Celery 事件推送和前端流式状态处理复杂度，超出本切片的“用户知道系统在做什么”目标。

5. **任务列表优先进入进度页，完成后保留结果入口**

   任务列表中 `created`、`queued`、`running`、`failed` 状态的主操作进入进度页；`completed` 状态继续保留商机推荐和基础报告入口，同时也可以提供进度/运行详情入口。新建研究提交成功后直接进入该任务进度页，避免用户回到列表后再寻找刚创建的任务。

## 风险 / 取舍

- [风险] 任务排队很快完成，用户可能只短暂看到进度页。 -> 缓解：完成后进度页仍展示已完成阶段和结果入口，让页面保留解释价值。
- [风险] 阶段事件写入失败但任务状态成功更新，时间线可能不完整。 -> 缓解：进度页同时展示任务当前阶段和状态；事件为空或缺失时展示中文说明，不阻塞完成后结果入口。
- [风险] 轮询增加少量 API 请求。 -> 缓解：只在 `queued` 或 `running` 状态轮询，完成或失败后停止。
- [风险] 失败事件的 `error_summary` 可能偏技术化。 -> 缓解：用户主文案使用任务 `failure_reason`，阶段事件错误摘要只做简短辅助展示，并保持不暴露堆栈或敏感信息。
- [风险] 任务列表操作变多导致界面拥挤。 -> 缓解：以“进度/查看”为主操作，完成状态才展示商机和报告入口，次要操作使用现有图标按钮和紧凑布局。

## 迁移计划

无需数据库迁移。上线步骤为：先补充后端进度读取 schema/API 和测试，再新增前端进度页与任务列表入口，最后调整新建研究成功跳转。回滚时可移除前端入口并保留新增读取接口，不影响现有任务创建、研究运行、商机列表、详情和基础报告。

## 开放问题

暂无阻塞问题。SSE 是否需要在后续切片实现，可以等 P0 进度页跑通后根据演示体验再决定。
