## 背景

MarketPilot 现在已经完成基础商机推荐、公开来源收集、来源透明度、需求洞察、货源候选、竞品参考和验证预算估算。用户可以看到“有没有需求”“去哪找货”“市场上有哪些参考”和“首批验证大概要花多少钱”，但还缺少“这个方向最容易在哪些环节踩坑、哪些风险需要先排除”的结构化判断。

`review-opportunity-risks` 是验证预算之后、行动计划之前的 P1 质量增强切片。它复用当前研究任务、商机、来源、需求洞察、货源候选、竞品参考、验证预算和 Agent 阶段事件能力，为每个商机生成可读、可追踪、仍需验证的风险复核，不改变 P0 基础商机生成边界。

## 目标 / 非目标

**目标：**

- 为每个商机生成并保存 1 条结构化风险复核。
- 覆盖产品质量、发货履约、售后、合规、库存积压、同质化竞争和平台规则风险。
- 输出综合风险等级、风险摘要、重点风险、分项风险说明、风险触发原因和可行动缓解建议。
- 优先使用商机基础字段、来源线索、需求洞察、货源候选、竞品参考和验证预算估算；缺少上下游数据时仍生成谨慎的待验证风险提示。
- 在商机详情展示完整风险复核，在基础报告展示风险摘要。
- 为风险复核阶段记录 Agent 运行事件、日志和 LangSmith trace 元数据。
- 本地和测试环境提供确定性回退，自动化测试不依赖外部 LLM、搜索或网络服务。

**非目标：**

- 不做法律、合规或供应链尽调结论，不声称已经确认产品合法性、平台准入、库存能力或供应商履约能力。
- 不自动联系供应商、自动检测平台规则、自动抓取登录后平台数据或自动下单。
- 不让风险复核反向修改商机排序、基础风险等级、价格带、利润粗算或推荐理由。
- 不生成完整验证行动计划、询盘话术或上架检查清单；这些由 `create-action-plan` 处理。
- 不引入 RAG、向量检索、LangSmith 数据集或自动评测；这些留给 `evaluate-rag-quality`。

## 技术决策

### 1. 新增 `opportunity_risks` 独立模型

风险复核作为商机结果的 P1 增强数据保存，不直接扩展 `opportunities` 表。建议新增 `opportunity_risks` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `research_task_id`: 内部外键，关联研究任务。
- `opportunity_id`: 内部外键，关联商机。
- `overall_risk_level`: `low`、`medium` 或 `high`，表达综合风险等级。
- `risk_summary`: 面向用户的风险摘要。
- `quality_risk`、`fulfillment_risk`、`after_sales_risk`、`compliance_risk`、`inventory_risk`、`competition_risk`、`platform_risk`: 分项风险说明，使用文本保留场景化判断。
- `risk_triggers`: JSON 字符串列表，记录触发风险判断的线索或假设。
- `mitigation_suggestions`: JSON 字符串列表，记录用户可以先做的缓解动作。
- `review_status`: `derived`、`fallback` 或 `insufficient_data`，表达风险复核依据状态。
- `created_at`、`updated_at`、`deleted_at`。

该表遵循项目数据库通用字段规范；默认读取只返回 `deleted_at` 为空的记录。重新运行研究任务并成功生成新商机后，旧商机关联的风险复核应被软删除或从默认读取中排除。

替代方案：把风险复核塞进 `opportunities` 的 JSON 字段。这样能少建表，但会让 P0 基础推荐和 P1 风险分析耦合，也不利于后续行动计划单独复用风险触发原因和缓解建议。

### 2. 不新增风险来源关联表

`research_sources` 已支持 `risk` 类型来源，第一版风险复核可以在生成上下文中读取现有来源摘要，但不新增 `opportunity_risk_sources` 关联表。风险复核保存 `risk_triggers` 和 `review_status`，用于表达“为什么提示这个风险”和“依据是否充分”。

如果后续需要把每个风险点绑定到具体来源，可以在单独 change 中增加关联表。当前保持模型简单，避免为了初步风险提示过早引入复杂证据链。

### 3. 风险复核生成放在验证预算之后

LangGraph 节点顺序调整为：

```text
normalize_intake
  -> generate_opportunities
  -> validate_results
  -> persist_results
  -> collect_research_sources
  -> generate_demand_insights
  -> generate_supply_candidates
  -> generate_competitor_references
  -> estimate_validation_budgets
  -> review_opportunity_risks
  -> END
```

生成节点读取当前任务的未软删除商机、可用来源线索、需求洞察、货源候选、竞品参考和验证预算估算。风险复核优先使用货源候选里的材质、起订量、供应商确认问题，竞品参考里的同质化程度和差异化建议，验证预算里的首批数量、预算和关键假设，以及来源线索里的风险类摘要；缺少任一输入时仍可基于任务输入和商机字段生成谨慎风险提示，并用 `fallback` 或 `insufficient_data` 标识。

替代方案：在竞品参考之后、验证预算之前生成风险复核。这样能更早产出风险，但会少用首批数量、预算假设和毛利不确定性，风险判断对“是否值得小预算验证”的帮助较弱。

### 4. 使用单次批量结构化 LLM 调用

单个任务最多 5 个商机，风险复核适合用一次批量 LLM 调用生成。输入包括任务上下文、商机基础字段、裁剪后的来源摘要、需求摘要、货源候选摘要、竞品参考摘要和验证预算摘要，输出为按商机 UUID 映射的 JSON。后端使用 Pydantic 校验字段、数量、枚举、文本长度和谨慎表达；校验失败可重试一次，最终失败时记录阶段失败。

本地和测试环境缺少 LLM key 时，使用确定性回退基于商机字段和现有增强数据生成稳定中文风险复核。生产环境缺少 LLM key 时按配置错误处理，但风险复核失败不影响基础结果。

替代方案：每个商机单独调用 LLM。隔离性更好，但成本和延迟更高，且 MVP 每次只有 3-5 个商机，批量更合适。

### 5. 风险复核失败不破坏基础结果

风险复核是 P1 增强能力。若商机生成、校验或持久化失败，任务仍按现有核心逻辑失败；若风险复核生成失败，任务保留 `completed`，已生成商机、来源、需求洞察、货源候选、竞品参考和验证预算仍可读取，`review_opportunity_risks` 阶段事件记录为失败，风险 API 返回空结果或空状态。

这样能避免 LLM 波动或上游数据不足破坏已完成的 P0 演示闭环，同时让演示和排障能通过阶段事件看到风险复核缺失原因。

替代方案：风险复核失败即整个研究任务失败。这样能保证完成任务都有风险复核，但会让 P1 增强能力影响核心闭环稳定性。

### 6. 前端只在结果页面内部展示

风险复核不新增一级导航。商机详情展示完整风险复核，包括综合风险等级、风险摘要、分项风险、触发原因和缓解建议。基础报告展示每个商机的风险摘要、最高优先级风险和进入详情的入口。缺少风险复核或读取失败时，页面保留基础商机内容，并展示中文非阻塞空状态。

文案继续使用“风险提示”“待验证”“需要确认”“建议先排查”等谨慎表达，不使用“合规已确认”“供应商履约已验证”“平台规则无风险”等结论式表达。

## 迁移计划

1. 新增数据库迁移，创建 `opportunity_risks`，包含 UUID、时间戳和软删除字段。
2. 新增后端模型、仓储、服务和 API，先保证空结果和 fallback 可用。
3. 将 `review_opportunity_risks` 接入研究任务阶段枚举、LangGraph 节点和前端阶段文案。
4. 增加前端读取函数和展示组件，接入商机详情与基础报告。
5. 补齐后端、前端和 OpenSpec 校验后再进入实现完成状态。

回滚时可以停止调用新节点并隐藏前端面板；新表作为 P1 增强数据保留，不影响已有基础商机、来源、需求洞察、货源候选、竞品参考或验证预算读取。

## 风险 / 取舍

- [风险] 用户误以为风险复核是正式尽调或合规结论 -> 缓解：规格、生成提示和 UI 均要求谨慎表达，并展示触发原因和待确认动作。
- [风险] 上游来源或候选数据不足导致风险提示泛化 -> 缓解：允许 `fallback` 和 `insufficient_data` 状态，明确风险是初步提示。
- [风险] 新增 LLM 节点增加耗时 -> 缓解：单次批量生成，限制输入摘要数量和文本长度。
- [风险] 风险复核失败影响演示节奏 -> 缓解：非阻塞失败，保留基础结果并记录阶段事件。
- [风险] 重新运行后旧风险复核污染新结果 -> 缓解：按任务或商机软删除旧风险复核，默认读取只返回当前未软删除数据。

## 开放问题

- 第一版是否需要为每个风险点单独记录严重度。建议第一版只保留综合风险等级和分项风险说明，避免 UI 和模型过早复杂化；后续如果行动计划需要排序风险点，再单独增强。
