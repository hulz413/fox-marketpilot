## 背景

MarketPilot 当前已经具备真实研究任务创建、启动运行、进度页、商机结果、报告、来源线索和需求洞察闭环。新建研究页已有默认中文表单值和 3 条中文示例需求，但示例按钮只填入自然语言 brief；任务列表空状态也只静态展示示例文本。演示人员仍需要进入表单、理解字段并手动提交，才能跑完一次真实演示。

本 change 要把这些示例变成可行动入口：用户从首次空状态或新建研究页点击示例，即可创建普通研究任务、启动基础商机研究，并进入该任务进度页。实现应复用现有 API 和运行链路，不新增静态 demo 结果，也不修改 Agent 生成逻辑。

## 目标 / 非目标

**目标：**

- 首次打开且没有任务时，用户能直接看到 2-3 个中文示例入口，并一键启动完整真实研究流程。
- 新建研究页继续支持示例填入，同时提供用示例直接创建并启动任务的能力。
- 示例任务保存为普通研究任务，后续可以在任务列表、历史、进度、商机、报告、来源和需求洞察中继续查看。
- 启动过程有明确 pending、错误和防重复点击体验。

**非目标：**

- 不新增后端 demo endpoint、seed 数据接口或专用 demo 表。
- 不引入静态商机结果、静态报告或预生成任务。
- 不改变基础商机研究、来源收集、需求洞察、LangSmith tracing 或 Celery worker 的执行语义。
- 不接入外部平台账号、登录态抓取或自动下单。

## 技术决策

1. **使用前端预置示例对象，而不是后端 demo 模板接口**

   示例数据放在现有前端产品数据附近，从纯字符串升级为结构化对象，包含 `brief`、`budget`、`target_channels`、`preferred_categories`、`excluded_categories`、`target_audience`、`expected_profit`、`supply_preferences` 和 `constraints`。这样示例既能填表，也能直接作为创建任务 payload。

   备选方案是新增后端模板接口，但当前示例数量少、内容偏演示文案，新增接口会扩大后端 surface，也需要额外测试和部署协调。

2. **一键启动复用现有创建和启动 API**

   点击示例后，前端依次调用已有任务创建 API 和启动运行 API。创建成功并启动后跳转 `/research/tasks/{taskUuid}`，让进度页继续承接运行状态。

   备选方案是新增“创建并启动”后端聚合接口。它能减少一次网络往返，但当前已有新建研究页已经采用前端串联调用，沿用现状更小、更一致。

3. **每次示例启动创建一条普通真实任务**

   示例任务不使用特殊 demo 标记，不复用隐藏任务，也不写入静态结果。任务列表和研究历史按已有规则展示，用户能重新运行或打开报告。

   备选方案是准备一条固定演示任务，但这会让演示结果与当前运行链路脱节，也容易和“不得展示静态 demo 结果作为真实结果”的既有约束冲突。

4. **共享启动逻辑并区分交互模式**

   新建研究页保留“点击示例填入表单”的低风险探索方式，同时提供“启动此示例”或等价主操作。空状态里的示例应直接启动，减少首次演示路径长度。相关按钮在 pending 时禁用，避免重复创建。

## 风险 / 取舍

- 示例点击会真实创建任务，误点可能产生多条研究记录 → 用明确按钮文案、pending 禁用和任务列表真实状态承接，避免误认为是静态预览。
- 前端串联两个 API 时可能创建成功但启动失败 → 保留中文错误提示，并允许用户进入已创建任务或从任务列表/进度页重新启动。
- 结构化示例对象会让产品数据略复杂 → 将字段集中在同一数据模块，并让填表和一键启动共用同一对象。
- 如果后端或 worker 未启动，首次演示会失败 → 使用现有错误处理文案说明 API 或后台任务服务不可用，不伪造演示结果。
