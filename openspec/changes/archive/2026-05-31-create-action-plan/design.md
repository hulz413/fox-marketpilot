## 背景

MarketPilot 现在已经完成基础商机推荐、公开来源收集、来源透明度、需求洞察、货源候选、竞品参考、验证预算估算和风险复核。用户可以看到“为什么值得试”“去哪找货”“市场上有哪些参考”“首批预算大概多少”和“哪些风险要先排查”，但还缺少一份能直接指导首轮小预算验证的执行计划。

`create-action-plan` 是 P1 执行建议切片。它复用当前研究任务、商机、来源、需求洞察、货源候选、竞品参考、验证预算、风险复核和 Agent 阶段事件能力，为每个商机生成可读、可追踪、仍需人工确认的行动计划，不改变 P0 基础商机生成边界。

## 目标 / 非目标

**目标：**

- 为每个商机生成并保存 1 条结构化行动计划。
- 覆盖首批验证计划、选品验证方式、内容种草角度、商品标题建议、卖点建议、供应商询盘话术和上架前检查清单。
- 优先使用商机基础字段、来源线索、需求洞察、货源候选、竞品参考、验证预算和风险复核；缺少上下游数据时仍生成谨慎的待验证行动建议。
- 在商机详情展示完整行动计划，在基础报告展示行动摘要。
- 为行动计划阶段记录 Agent 运行事件、日志和 LangSmith trace 元数据。
- 本地和测试环境提供确定性回退，自动化测试不依赖外部 LLM、搜索或网络服务。

**非目标：**

- 不自动下单、自动联系供应商、自动发布内容或自动创建平台商品。
- 不接入登录后平台数据、供应商账号、电商平台账号或内容平台账号。
- 不承诺成交、转化、利润、回本、供应商履约或平台审核通过。
- 不让行动计划反向修改商机排序、基础风险等级、价格带、预算估算或风险复核。
- 不引入 RAG、向量检索、LangSmith 数据集或自动评测；这些留给 `evaluate-rag-quality`。

## 技术决策

### 1. 新增 `action_plans` 独立模型

行动计划作为商机结果的 P1 增强数据保存，不直接扩展 `opportunities` 表。建议新增 `action_plans` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `research_task_id`: 内部外键，关联研究任务。
- `opportunity_id`: 内部外键，关联商机。
- `validation_goal`: 首轮验证目标，说明这次小预算验证要确认什么。
- `first_batch_plan`: 首批验证计划，包含建议数量、预算约束、试跑方式或节奏。
- `product_validation_method`: 选品验证方式，说明样品、小单、反馈或询单怎么验证。
- `content_angles`: JSON 字符串列表，记录内容种草角度。
- `title_suggestions`: JSON 字符串列表，记录商品标题或内容标题建议。
- `selling_point_suggestions`: JSON 字符串列表，记录可测试卖点。
- `supplier_inquiry_script`: 供应商询盘话术，供用户复制后人工调整。
- `prelaunch_checklist`: JSON 字符串列表，记录上架前检查事项。
- `plan_status`: `derived`、`fallback` 或 `insufficient_data`，表达行动计划依据状态。
- `created_at`、`updated_at`、`deleted_at`。

该表遵循项目数据库通用字段规范；默认读取只返回 `deleted_at` 为空的记录。重新运行研究任务并成功生成新商机后，旧商机关联的行动计划应被软删除或从默认读取中排除。

替代方案：把行动计划塞进 `opportunities` 的 JSON 字段。这样能少建表，但会让 P0 基础推荐和 P1 执行建议耦合，也不利于后续单独重跑或展示行动计划。

### 2. 不新增来源关联表

行动计划第一版是对已有结构化分析的执行层汇总，主要依据来自商机字段、货源候选、竞品参考、验证预算和风险复核，而不是直接新增外部来源。因此第一版只保存 `plan_status` 和结构化行动字段，不新增 `action_plan_sources`。

如果后续需要把每个行动建议绑定到具体来源，可以在单独 change 中增加关联表。当前保持模型简单，让行动计划先服务 MVP 演示闭环。

### 3. 行动计划生成放在风险复核之后

LangGraph 节点顺序调整为：

```text
normalize_intake
  -> generate_opportunities
  -> validate_results
  -> persist_results
  -> collect_research_sources
  -> generate_demand_insights
  -> generate_supply_candidates
  -> generate_competitor_references
  -> estimate_validation_budgets
  -> review_opportunity_risks
  -> create_action_plans
  -> END
```

生成节点读取当前任务的未软删除商机、可用来源线索、需求洞察、货源候选、竞品参考、验证预算估算和风险复核。行动计划优先使用货源候选里的供应商待确认问题、竞品参考里的差异化角度、验证预算里的首批数量和关键假设，以及风险复核里的触发原因和缓解建议；缺少任一输入时仍可基于任务输入和商机字段生成谨慎执行建议，并用 `fallback` 或 `insufficient_data` 标识。

替代方案：在验证预算之后、风险复核之前生成行动计划。这样能更早产出执行步骤，但会少用风险复核中的质量、履约、售后、合规和平台规则提示，检查清单质量较弱。

### 4. 使用单次批量结构化 LLM 调用

单个任务最多 5 个商机，行动计划适合用一次批量 LLM 调用生成。输入包括任务上下文、商机基础字段、裁剪后的来源摘要、需求摘要、货源候选摘要、竞品参考摘要、验证预算摘要和风险摘要，输出为按商机 UUID 映射的 JSON。后端使用 Pydantic 校验字段、数量、枚举、列表长度、文本长度和谨慎表达；校验失败可重试一次，最终失败时记录阶段失败。

本地和测试环境缺少 LLM key 时，使用确定性回退基于商机字段和现有增强数据生成稳定中文行动计划。生产环境缺少 LLM key 时按配置错误处理，但行动计划失败不影响基础结果。

替代方案：每个商机单独调用 LLM。隔离性更好，但成本和延迟更高，且 MVP 每次只有 3-5 个商机，批量更合适。

### 5. 行动计划失败不破坏基础结果

行动计划是 P1 增强能力。若商机生成、校验或持久化失败，任务仍按现有核心逻辑失败；若行动计划生成失败，任务保留 `completed`，已生成商机、来源、需求洞察、货源候选、竞品参考、验证预算和风险复核仍可读取，`create_action_plans` 阶段事件记录为失败，行动计划 API 返回空结果或空状态。

这样能避免 LLM 波动或上游数据不足破坏已完成的 P0 演示闭环，同时让演示和排障能通过阶段事件看到行动计划缺失原因。

替代方案：行动计划失败即整个研究任务失败。这样能保证完成任务都有执行建议，但会让 P1 增强能力影响核心闭环稳定性。

### 6. 前端只在结果页面内部展示

行动计划不新增一级导航。商机详情展示完整行动计划，包括验证目标、首批计划、验证方式、内容角度、标题建议、卖点建议、询盘话术和上架前检查清单。基础报告展示每个商机的行动摘要和进入详情的入口。缺少行动计划或读取失败时，页面保留基础商机内容，并展示中文非阻塞空状态。

文案继续使用“验证计划”“建议先做”“待确认”“人工调整后使用”等谨慎表达，不使用“保证成交”“保证回本”“供应商已确认”“上架必过”等结论式表达。

## 迁移计划

1. 新增数据库迁移，创建 `action_plans`，包含 UUID、时间戳和软删除字段。
2. 新增后端模型、仓储、服务和 API，先保证空结果和 fallback 可用。
3. 将 `create_action_plans` 接入研究任务阶段枚举、LangGraph 节点和前端阶段文案。
4. 增加前端读取函数和展示组件，接入商机详情与基础报告。
5. 补齐后端、前端和 OpenSpec 校验后再进入实现完成状态。

回滚时可以停止调用新节点并隐藏前端面板；新表作为 P1 增强数据保留，不影响已有基础商机、来源、需求洞察、货源候选、竞品参考、验证预算或风险复核读取。

## 风险 / 取舍

- [风险] 用户误以为询盘话术会被系统自动发送 -> 缓解：规格和 UI 明确话术仅供复制和人工调整，不自动联系供应商。
- [风险] 行动计划过于泛化，缺少可执行性 -> 缓解：生成上下文纳入货源候选、预算假设和风险缓解建议，并要求输出检查清单和询盘话术。
- [风险] 新增 LLM 节点增加耗时 -> 缓解：单次批量生成，限制输入摘要数量和文本长度。
- [风险] 行动计划失败影响演示节奏 -> 缓解：非阻塞失败，保留基础结果并记录阶段事件。
- [风险] 重新运行后旧行动计划污染新结果 -> 缓解：按任务或商机软删除旧行动计划，默认读取只返回当前未软删除数据。

## 开放问题

- 第一版是否需要把行动计划拆成多日任务排期。建议第一版不做日期排期，只输出首批验证计划和检查清单，避免过早进入项目管理工具形态。
