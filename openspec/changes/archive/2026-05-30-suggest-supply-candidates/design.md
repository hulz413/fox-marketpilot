## 背景

MarketPilot 现在已经完成基础商机推荐、公开来源收集、来源透明度和需求洞察。用户可以理解“为什么这个商机可能有需求”，但还缺少“我该去哪里找货、搜什么款、价格和起订量大概如何、询盘时要问什么”的结构化建议。

`suggest-supply-candidates` 是需求洞察之后的 P1 质量增强切片。它复用当前研究任务、商机、来源和 Agent 阶段事件能力，为每个商机生成可读、可追踪、仍需验证的货源候选，不改变 P0 基础商机生成边界。

## 目标 / 非目标

**目标：**

- 为每个商机生成并保存至少 2 个结构化货源候选方向或候选商品。
- 覆盖供给市场、搜索关键词、价格区间、起订量、规格/材质/款式参考和供应商确认问题。
- 优先关联现有 `supply` 和 `general` 来源线索，缺少来源时仍生成谨慎的待验证货源建议。
- 在商机详情展示完整货源候选，在基础报告展示摘要。
- 为货源候选生成阶段记录 Agent 运行事件、日志和 LangSmith trace 元数据。
- 本地和测试环境提供确定性回退，自动化测试不依赖外部 LLM、搜索或网络服务。

**非目标：**

- 不自动下单、不自动联系供应商、不确认真实库存、报价有效期或履约能力。
- 不抓取登录后平台数据，不绕过平台访问限制。
- 不做竞品参考、利润预算、风险复核或完整行动计划；这些由后续 P1 切片处理。
- 不让货源候选反向修改商机排序、风险等级、价格带或基础推荐理由。
- 不引入 RAG、向量检索、LangSmith 数据集或自动评测；这些留给 `evaluate-rag-quality`。
- 不把候选货源表达为“已确认供给”“已核验供应商”或等价结论。

## 技术决策

### 1. 分析类表使用短表名

需求洞察和货源候选都是商机下的 P1 分析结果，但归属关系已经由 `research_task_id` 和 `opportunity_id` 字段表达，表名不再重复增加 `opportunity_` 前缀。统一命名为：

- `demand_insights`
- `demand_insight_sources`
- `supply_candidates`
- `supply_candidate_sources`

当前已实现的需求洞察表在本 change 实现时同步改为短表名。若旧迁移已经在共享环境执行过，应使用新迁移执行 rename；若仍只在本地开发环境使用，可以在实现时按项目迁移策略决定是否重写历史迁移。

替代方案：继续使用带 `opportunity_` 前缀的长表名。这样语义更显式，但表名偏长，且与字段里的 `opportunity_id` 重复。

### 2. 新增 `supply_candidates` 独立模型

货源候选作为商机结果的 P1 增强数据保存，不直接扩展 `opportunities` 表。建议新增 `supply_candidates` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `research_task_id`: 内部外键，关联研究任务。
- `opportunity_id`: 内部外键，关联商机。
- `rank`: 同一商机下的候选排序。
- `candidate_name`: 候选货源方向或候选商品名称。
- `supply_market`: 可能的供给市场，例如 `1688`、义乌小商品批发、国内公开供给市场。
- `search_keywords`: JSON 字符串列表，用于继续搜索或询盘。
- `price_range`: 参考价格区间。
- `minimum_order_quantity`: 起订量或混批说明。
- `specification_notes`: JSON 字符串列表，规格、材质、款式、包装等参考。
- `supplier_questions`: JSON 字符串列表，需要向供应商确认的问题。
- `recommendation_note`: 为什么该候选适合当前商机的说明。
- `source_status`: `linked`、`no_sources` 或 `fallback`。
- `created_at`、`updated_at`、`deleted_at`。

该表遵循项目数据库通用字段规范；默认读取只返回 `deleted_at` 为空的记录。重新运行研究任务并成功生成新商机后，旧商机关联的货源候选应被软删除或从默认读取中排除。

替代方案：直接把候选货源塞进 `opportunities` 的 JSON 字段。这样能少建表，但会让 P0 基础结果和 P1 分析结果耦合，也不利于后续预算、风险和行动计划复用货源候选。

### 3. 使用关联表连接货源候选和研究来源

货源候选需要说明哪些公开线索可作为初步参考。建议新增 `supply_candidate_sources` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `supply_candidate_id`: 内部外键，关联货源候选。
- `research_source_id`: 内部外键，关联研究来源。
- `relevance_note`: 该来源与候选货源的关联说明。
- `created_at`、`updated_at`、`deleted_at`。

API 返回时暴露来源 UUID、标题、URL、摘要、支撑强度和关联说明，不暴露内部自增 ID。这样保持来源数据层稳定，也能让后续预算或风险切片复用同一候选货源及其来源关系。

替代方案：在货源候选表中保存来源 UUID 的 JSON 数组。实现更快，但缺少关系完整性，也不利于按来源反查分析结论。

### 4. 货源候选生成放在需求洞察之后

LangGraph 节点顺序调整为：

```text
normalize_intake
  -> generate_opportunities
  -> validate_results
  -> persist_results
  -> collect_research_sources
  -> generate_demand_insights
  -> generate_supply_candidates
  -> END
```

生成节点读取当前任务的未软删除商机、来源和可用需求洞察。每个商机优先使用与该商机关联且类型为 `supply` 或 `general` 的来源；如果没有专属来源，可以使用任务级 `supply` 或 `general` 来源；仍没有来源时，基于任务输入、供给来源偏好和商机字段生成谨慎的待验证货源候选，并将 `source_status` 标记为 `no_sources` 或 `fallback`。

替代方案：在商机生成前先做货源搜索，再生成推荐。这个路径更接近完整研究 Agent，但会改变当前“先跑通基础推荐，再补强证据”的边界和失败模式，范围过大。

### 5. 使用单次批量结构化 LLM 调用

单个任务最多 5 个商机，货源候选适合用一次批量 LLM 调用生成。输入包括任务上下文、商机基础字段、供给来源偏好和裁剪后的供给来源摘要，输出为按商机 UUID 映射的 JSON。后端使用 Pydantic 校验字段、数量、排序、文本长度和谨慎表达；校验失败可重试一次，最终失败时记录阶段失败。

本地和测试环境缺少 LLM key 时，使用确定性回退基于商机字段和来源摘要生成稳定中文候选。生产环境缺少 LLM key 时按配置错误处理，但货源候选失败不影响基础结果。

替代方案：每个商机单独调用 LLM。隔离性更好，但成本和延迟更高，且 MVP 每次只有 3-5 个商机，批量更合适。

### 6. 货源候选失败不破坏基础结果

货源候选是 P1 增强能力。若商机生成、校验或持久化失败，任务仍按现有核心逻辑失败；若货源候选生成失败，任务保留 `completed`，已生成商机、来源和需求洞察仍可读取，`generate_supply_candidates` 阶段事件记录为失败，货源候选 API 返回空结果或空状态。

这样能避免 LLM 波动或来源不足破坏已完成的 P0 演示闭环，同时让演示和排障能通过阶段事件看到候选缺失原因。

替代方案：货源候选失败即整个研究任务失败。这样能保证完成任务都有货源候选，但会让 P1 增强能力影响核心闭环稳定性。

### 7. 前端只在结果页面内部展示

货源候选不新增一级导航。商机详情展示完整候选列表，包括候选名称、供给市场、价格区间、起订量、规格参考、确认问题和关联来源。基础报告展示每个商机的货源候选摘要和进入详情的入口。缺少候选、来源为空或读取失败时，页面保留基础商机内容，并展示中文非阻塞空状态。

文案继续使用“候选”“初步参考”“需要确认”“待验证”等谨慎表达，不使用“已确认供给”“供应商已核验”等结论式表达。

## 迁移计划

1. 统一分析类表短命名，将需求洞察相关表调整为 `demand_insights` 和 `demand_insight_sources`。
2. 新增数据库迁移，创建 `supply_candidates` 和 `supply_candidate_sources`，包含 UUID、时间戳和软删除字段。
3. 新增后端模型、仓储、服务和 API，先保证空结果和 fallback 可用。
4. 将 `generate_supply_candidates` 接入研究任务阶段枚举、LangGraph 节点和前端阶段文案。
5. 增加前端读取函数和展示组件，接入商机详情与基础报告。
6. 补齐后端、前端和 OpenSpec 校验后再进入实现完成状态。

回滚时可以停止调用新节点并隐藏前端面板；新表作为 P1 增强数据保留，不影响已有基础商机读取。

## 风险 / 取舍

- [风险] 用户误以为候选货源已经核验 -> 缓解：规格、生成提示和 UI 均要求谨慎表达，并展示来源状态和确认问题。
- [风险] 公开来源不足导致候选泛化 -> 缓解：允许 `no_sources` 状态，明确候选是基于任务输入和商机字段的待验证方向。
- [风险] 新增 LLM 节点增加耗时 -> 缓解：单次批量生成，限制输入来源数量和文本长度。
- [风险] 货源候选失败影响演示节奏 -> 缓解：非阻塞失败，保留基础结果并记录阶段事件。
- [风险] 重新运行后旧候选污染新结果 -> 缓解：按任务或商机软删除旧候选，默认读取只返回当前未软删除数据。

## 开放问题

- 第一版是否需要展示具体外链商品，还是只展示候选方向和搜索关键词。建议第一版允许来源链接作为参考，但不把单个链接包装成已确认商品。
