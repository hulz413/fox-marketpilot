## 为什么

当前 MarketPilot 已经能生成基础商机、收集公开来源并解释需求洞察，但用户还不知道这些商机大致该去哪里找货、该看哪些候选商品、价格和起订量是否适合小预算验证。下一步需要把“推荐一个产品方向”推进到“可以开始找货和询盘”的可执行判断。

## 变更内容

- 为每个已生成商机补充货源候选，覆盖候选货源方向、可能的供给市场、价格区间、起订量、规格/材质/款式参考和供应商确认问题。
- 优先使用现有 `supply` 或 `general` 来源线索，并在缺少来源时生成谨慎的待验证货源建议。
- 在研究运行完成来源收集和需求洞察后生成货源候选，并保存为可读取的结构化结果。
- 在商机详情和基础报告中展示货源候选摘要、来源状态、确认问题和缺失来源时的非阻塞说明。
- 提供本地和测试环境确定性回退，确保自动化测试不依赖外部 LLM、搜索或网络服务。

## 能力

### 新增能力

- `supply-candidates`: 定义每个商机的货源候选生成、持久化、读取、来源关联、前端展示和非阻塞失败契约。

### 修改能力

- `opportunity-research`: 研究运行在需求洞察之后生成货源候选，并记录对应阶段事件；货源候选失败不破坏已完成的基础商机结果。
- `research-progress`: 进度页需要展示货源候选生成阶段和对应的谨慎中文说明。
- `agent-run-observability`: Agent 阶段历史和 trace 需要覆盖货源候选生成阶段，便于解释候选缺失或生成失败。

## 影响

- 后端新增货源候选领域模型、迁移、模式、仓储、服务和读取 API。
- 数据库分析类表统一使用短表名：`demand_insights`、`demand_insight_sources`、`supply_candidates`、`supply_candidate_sources`，归属关系由 `research_task_id` 和 `opportunity_id` 表达。
- LangGraph 研究流程新增货源候选生成阶段，并复用已有研究任务、商机、来源和 Agent 运行事件能力。
- LLM 生成逻辑增加货源候选结构化输出；本地和测试环境增加确定性回退。
- 前端商机详情和基础报告增加货源候选展示区块，并保持缺少来源或候选时的非阻塞中文空状态。
- 测试覆盖货源候选生成、读取、来源关联、重新运行替换、运行阶段事件和前端 UI 契约。
