## 背景

MarketPilot 已完成 P0 研究闭环和 P1 来源透明度基础：研究任务可以生成 3-5 个待验证商机，保存公开来源线索，并在报告和详情中展示来源列表。当前商机详情仍主要依赖 `recommendation_reason` 和 `next_step_summary` 两段文本，缺少对需求判断的结构化拆解。

`explain-demand-insights` 是来源透明度之后的第一个深度分析切片。它需要复用现有商机、研究任务、来源和 Agent 运行事件能力，为每个商机生成“为什么可能有需求”的可读证据链，同时保持“待验证”语气，不改变 P0 基础商机生成边界。

## 目标 / 非目标

**目标：**

- 为每个商机生成并保存结构化需求洞察。
- 覆盖目标人群、使用场景、购买动机、内容种草点、趋势信号和季节性因素。
- 优先关联现有 `demand` 和 `general` 来源线索，缺少来源时仍生成谨慎的待验证洞察。
- 在商机详情展示完整需求洞察，在基础报告展示摘要。
- 为需求洞察生成阶段记录 Agent 运行事件、日志和 LangSmith trace 元数据。
- 本地和测试环境提供确定性回退，自动化测试不依赖外部 LLM 或网络服务。

**非目标：**

- 不引入 RAG、向量检索、LangSmith 数据集或自动评测；这些留给 `evaluate-rag-quality`。
- 不做货源候选、竞品参考、利润预算、风险复核或完整行动计划；这些由后续 P1 切片处理。
- 不让需求洞察反向修改商机排序、风险等级、价格带或基础推荐理由。
- 不抓取登录后平台数据，不联系供应商，不自动发布内容。
- 不把来源或洞察表达为“已证明有市场”。

## 技术决策

### 1. 新增 `opportunity_demand_insights` 独立模型

需求洞察作为商机结果的 P1 增强数据保存，不直接扩展 `opportunities` 表的大字段。建议新增 `opportunity_demand_insights` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `research_task_id`: 内部外键，关联研究任务。
- `opportunity_id`: 内部外键，关联商机。
- `summary`: 需求洞察摘要。
- `audience_profile`: 更具体的人群画像。
- `use_cases`: JSON 字符串列表，使用场景。
- `purchase_motivations`: JSON 字符串列表，购买动机。
- `content_angles`: JSON 字符串列表，内容种草角度。
- `trend_signals`: JSON 字符串列表，趋势或需求信号。
- `seasonality_notes`: 季节性或时点因素说明。
- `source_status`: `linked`、`no_sources` 或 `fallback`。
- `created_at`、`updated_at`、`deleted_at`。

该表遵循项目数据库通用字段规范；默认读取只返回 `deleted_at` 为空的记录。重新运行研究任务并成功生成新商机后，旧商机关联的需求洞察应被软删除或从默认读取中排除。

替代方案：直接把需求洞察字段加入 `opportunities` 表。这样读取简单，但会把 P0 基础结果和 P1 分析结果耦合，后续货源、竞品、风险等切片也会继续膨胀该表。

### 2. 使用关联表连接需求洞察和研究来源

需求洞察需要说明哪些来源可能支持对应判断。建议新增 `opportunity_demand_insight_sources` 表：

- `id`: 内部自增主键。
- `uuid`: 对外公开 UUID。
- `demand_insight_id`: 内部外键，关联需求洞察。
- `research_source_id`: 内部外键，关联研究来源。
- `relevance_note`: 该来源支持洞察的说明。
- `created_at`、`updated_at`、`deleted_at`。

API 返回时暴露来源 UUID、标题、URL、摘要、支撑强度和关联说明，不暴露内部自增 ID。这样保持来源数据层稳定，也避免把来源 UUID 的 JSON 列表散落在洞察记录里。

替代方案：在需求洞察表中保存来源 UUID 的 JSON 数组。实现更快，但缺少关系完整性，也不利于后续按来源反查分析结论。

### 3. 需求洞察生成放在来源收集之后

LangGraph 节点顺序调整为：

```text
normalize_intake
  -> generate_opportunities
  -> validate_results
  -> persist_results
  -> collect_research_sources
  -> generate_demand_insights
  -> END
```

生成节点读取当前任务的未软删除商机和来源。每个商机优先使用与该商机关联且类型为 `demand` 或 `general` 的来源；如果没有专属来源，可以使用任务级 `demand` 或 `general` 来源；仍没有来源时，基于任务输入和商机字段生成谨慎的待验证洞察，并将 `source_status` 标记为 `no_sources`。

替代方案：在商机生成前先分析需求，再生成推荐。这个路径更像完整研究 Agent，但会改变 P0 推荐生成边界和失败模式，范围过大。

### 4. 使用单次批量结构化 LLM 调用

单个任务最多 5 个商机，需求洞察适合用一次批量 LLM 调用生成。输入包括任务上下文、商机基础字段和裁剪后的需求来源摘要，输出为按商机 UUID 映射的 JSON。后端使用 Pydantic 校验字段、数量和文本长度；校验失败可重试一次，最终失败时记录阶段失败。

本地和测试环境缺少 LLM key 时，使用确定性回退基于商机字段和来源摘要生成稳定中文洞察。生产环境缺少 LLM key 时按配置错误处理，但不暴露敏感信息。

替代方案：每个商机单独调用 LLM。隔离性更好，但成本和延迟更高，且 MVP 每次只有 3-5 个商机，批量更合适。

### 5. 需求洞察失败不破坏基础结果

需求洞察是 P1 增强能力。若商机生成、校验或持久化失败，任务仍按现有核心逻辑失败；若需求洞察生成失败，任务保留 `completed`，已生成商机和来源仍可读取，`generate_demand_insights` 阶段事件记录为失败，需求洞察 API 返回空结果或空状态。

这样能避免 LLM 波动破坏已完成的 P0 演示闭环，同时让演示和排障能通过阶段事件看到需求洞察失败原因。

替代方案：需求洞察失败即整个研究任务失败。这样能保证完成任务都有洞察，但会让 P1 增强能力影响核心闭环稳定性。

### 6. 前端只在结果页面内部展示

需求洞察不新增一级导航。商机详情展示完整洞察区块，基础报告展示每个商机的需求摘要和入口。缺少洞察、来源为空或读取失败时，页面保留基础商机内容，并展示中文非阻塞空状态。

文案继续使用“可能”“初步参考”“待验证”等谨慎表达，不使用“已证明”“确定有市场”等结论式表达。

## 风险 / 取舍

- [风险] 需求洞察听起来像已验证市场结论 -> 缓解：规格和 UI 均要求谨慎表达，并展示来源状态。
- [风险] 来源不足导致洞察质量有限 -> 缓解：允许 `no_sources` 状态，明确它是基于任务输入和基础商机字段的待验证判断。
- [风险] 新增 LLM 节点增加耗时 -> 缓解：单次批量生成，限制输入来源数量和文本长度。
- [风险] 需求洞察失败影响演示节奏 -> 缓解：非阻塞失败，保留基础结果并记录阶段事件。
- [风险] 重新运行后旧洞察污染新结果 -> 缓解：按任务或商机软删除旧洞察，默认读取只返回当前未软删除数据。
