## 为什么

当前 P0 闭环已经能从创建研究任务走到商机列表、详情和基础报告，但页面仍像功能骨架：全局导航会进入无上下文空状态，任务列表操作分散，移动端首屏被侧边栏占满，用户不容易判断下一步该看进度、结果还是报告。

本 change 聚焦打磨 MarketPilot 作为商机顾问 Agent 的核心 UI/UX，让研究任务成为稳定上下文，把进度、商机、报告和来源入口串成更顺手的工作台体验。

## 变更内容

- 将全局导航和任务内导航分层：全局入口负责开始和查找任务；进入任务后用任务上下文导航连接进度、商机推荐、基础报告和来源。
- 优化研究任务列表：减少重复标题信息，按任务状态突出一个主操作，并将 LangSmith 等调试入口降级为次要入口。
- 优化进度页：突出当前阶段、完成度和下一步操作，把 run ID、trace 等技术信息放入调试区域。
- 优化商机列表、详情和报告：从表格骨架升级为可决策的结果阅读体验，突出推荐理由、风险、验证动作和任务上下文。
- 使用已存在的来源读取能力，在报告或详情中展示来源入口、来源空状态和谨慎说明，不宣称已完成完整市场或竞品核验。
- 优化小屏体验：移动端不让侧边栏占满首屏；宽表格在手机端改为卡片式信息展示或可理解的列表布局。
- 不修改 Agent 生成逻辑、商机排序算法、数据库表结构、外部搜索策略或 LangSmith tracing 接入。

## 能力

### 新增能力

- 无

### 修改能力

- `product-skeleton`: 调整产品壳、全局导航、任务内导航、空状态和响应式布局的用户体验要求。
- `research-tasks`: 调整任务列表的信息层级、状态相关主操作和历史/真实任务入口行为。
- `research-progress`: 调整进度页的阶段摘要、任务上下文导航、调试信息和状态相关操作展示。
- `opportunity-results`: 调整商机列表、商机详情和基础报告的可决策信息展示与任务上下文衔接。
- `research-sources`: 调整已收集来源在前端结果页中的轻量展示、空状态和谨慎表达要求。

## 影响

- 前端应用壳、导航、任务列表、进度页、商机列表、商机详情、基础报告和历史页。
- 前端数据请求可能需要复用现有任务、商机、进度和来源 API；不新增后端存储模型。
- UI 自动化或手动验收需要覆盖桌面、窄屏和手机宽度下的主流程。
- 现有后端 API、数据库迁移、Agent 编排、Celery worker、LangSmith tracing 和来源收集逻辑不属于本 change 的实现范围。
