## ADDED Requirements

### Requirement: 研究任务列表采用状态驱动主操作

前端 SHALL 在研究任务列表中按任务状态突出一个最重要的下一步操作，并降低诊断入口和次要入口的视觉权重。

#### Scenario: 查看已完成任务

- **WHEN** 用户在研究任务列表查看状态为 `completed` 的任务
- **THEN** 列表项展示查看结果作为主操作
- **AND** 用户可以继续进入该任务的商机推荐或基础报告
- **AND** 进度页和 LangSmith trace 入口可以作为次要操作展示

#### Scenario: 查看运行中任务

- **WHEN** 用户在研究任务列表查看状态为 `queued` 或 `running` 的任务
- **THEN** 列表项展示查看进度作为主操作
- **AND** 列表项展示当前阶段的中文摘要
- **AND** 列表项不展示不可用的商机或报告主操作

#### Scenario: 查看可启动或失败任务

- **WHEN** 用户在研究任务列表查看状态为 `created` 或 `failed` 的任务
- **THEN** 列表项展示启动研究或重新运行作为主操作
- **AND** 失败任务展示中文失败摘要或进入进度页查看失败原因的入口
- **AND** 列表项不要求用户先进入商机列表才能恢复任务

### Requirement: 研究任务列表减少重复和噪音

前端 SHALL 在研究任务列表中避免重复展示同一段需求，并让用户快速扫视任务差异。

#### Scenario: 任务标题和需求相同

- **WHEN** 任务标题与自然语言需求内容相同或高度重复
- **THEN** 列表项只展示一次主要文本
- **AND** 次要文本展示预算、渠道、目标人群、创建时间或当前阶段等有区分度信息
- **AND** 文本过长时在视觉上截断但不破坏布局

#### Scenario: 任务有 trace URL

- **WHEN** 任务包含 LangSmith trace URL
- **THEN** 列表项可以提供 trace 外链
- **AND** trace 外链标识为调试或运行详情入口
- **AND** trace 外链不得抢占用户查看结果或进度的主操作位置

### Requirement: 研究历史使用真实任务入口

前端 SHALL 让研究历史入口复用真实研究任务数据或明确说明暂不可用，不展示会误导用户的静态历史结果。

#### Scenario: 查看研究历史

- **WHEN** 用户打开研究历史页面
- **THEN** 页面展示真实研究任务或明确的中文空状态
- **AND** 已完成任务可以进入对应真实报告
- **AND** 运行中或失败任务可以进入对应真实进度页

#### Scenario: 重新运行历史任务

- **WHEN** 用户从历史页面重新运行一条任务
- **THEN** 系统使用该任务已保存的研究需求和条件
- **AND** 用户进入该任务进度页查看新的运行状态
- **AND** 前端不只跳转到带默认示例的新建研究页
