## 背景

MarketPilot 当前前端已经具备浅色企业应用壳、新建研究、任务列表、进度页、商机列表、商机详情、基础报告和历史页。实际浏览后，主要问题不是缺少页面，而是用户上下文断裂：全局导航会把用户带到缺少任务上下文的页面，完成任务后的结果入口分散，表格在小屏依赖横向滚动，报告与详情仍偏“字段展示”而不是“商机决策辅助”。

本 change 不涉及数据库表、软删除读取规则或 Agent 执行链路变更。现有任务、商机、进度和来源 API 继续只返回未软删除记录，并继续使用 uuid 作为前端路由和跨模块关联标识。

## 目标 / 非目标

**目标：**

- 让研究任务成为结果浏览的主上下文，用户能从一个任务稳定进入进度、商机、报告和来源。
- 降低任务列表和结果页的认知负担，突出每个状态下最合理的主操作。
- 改善移动端和窄屏体验，避免侧边栏和宽表格阻断主流程。
- 在不夸大结论的前提下，把已收集来源作为“公开线索/初步参考”展示给用户。
- 保持默认中文文案，并让布局和组件结构可继续承接后续中英文 UI 切换。

**非目标：**

- 不新增或修改数据库表、迁移、后端核心业务模型。
- 不改变 Agent 生成逻辑、商机排序、来源收集策略、LangSmith tracing 或 Celery worker 行为。
- 不实现完整 RAG、深度竞品核验、精确利润模型、自动下单或自动联系供应商。
- 不把来源表达为已证明结论；来源仍是初步线索。

## 技术决策

1. **采用任务上下文导航，而不是继续依赖全局结果入口。**
   - 进入单个任务相关页面后，页面顶部提供同一任务的「进度」「商机」「报告」「来源」入口。
   - 全局「商机推荐」「最终报告」入口只负责引导用户选择最近已完成任务或返回任务列表，避免 `/reports/demo-report` 这类无真实上下文入口。
   - 备选方案是保留所有现有全局入口并增强空状态，但这仍要求用户自己理解任务上下文，不能解决“不顺手”的根因。

2. **任务列表改为状态驱动的操作模型。**
   - 每条任务展示标题、简短需求、状态、当前阶段、创建时间和一个主操作。
   - `completed` 的主操作是查看结果，`queued/running` 是查看进度，`created/failed` 是启动或重新运行；进度、报告、LangSmith 等放入次要操作。
   - 备选方案是继续显示多个并列按钮，但这会让列表密度变高且主次不清。

3. **移动端使用卡片化结果，不强迫用户横向滚动宽表。**
   - 桌面端保留高密度表格；小屏任务列表、商机列表和历史列表使用卡片或简化列表展示核心字段和主操作。
   - 备选方案是只包一层 `overflow-x-auto`，实现成本低，但手机上用户很难比较和操作。

4. **来源展示复用现有读取能力，作为轻量信任层。**
   - 报告和详情页可以读取任务或商机关联来源，展示来源数量、类型、摘要、链接和空状态。
   - 来源区域文案必须使用“公开线索”“初步参考”“待验证”等谨慎表达。
   - 备选方案是等完整来源透明度 change 再做，但当前已有来源 API，轻量展示能明显改善信任感。

5. **调试信息与用户决策信息分区。**
   - run ID、trace URL、阶段事件细节仍保留，但放入调试/运行详情区域。
   - 商机名称、推荐理由、风险、验证动作、来源线索放在用户决策区域。
   - 备选方案是继续把技术字段放在首屏卡片内，但会让非开发用户分心。

## 风险 / 取舍

- [Risk] UI 改动横跨多个页面，容易出现不一致的任务上下文链接。→ 通过共享任务上下文导航组件和端到端主流程验收降低风险。
- [Risk] 来源数据可能为空或质量不稳定，直接展示会削弱信任。→ 空状态明确说明来源暂不可用，并保持基础商机结果可读。
- [Risk] 卡片化移动端列表会降低桌面信息密度。→ 桌面保留表格，小屏才切换卡片。
- [Risk] 全局报告入口改为任务选择可能让老入口看起来少一步。→ 提供最近已完成任务快捷入口，并保留返回任务列表的清晰路径。
- [Risk] 文案如果过度包装会暗示已完成市场核验。→ 规格要求所有来源和基础报告文案保持待验证语气。
