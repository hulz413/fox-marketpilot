## ADDED Requirements

### Requirement: 系统可以创建和读取研究需求对齐会话

系统 SHALL 允许用户创建研究需求对齐会话，用于在正式研究任务创建前保存聊天上下文、结构化草稿和对齐状态。

#### Scenario: 创建空白对齐会话

- **WHEN** 用户打开聊天式新建研究入口并创建会话
- **THEN** 系统创建一条研究需求对齐会话
- **AND** 会话包含公开 UUID、状态、空草稿、空缺失字段列表、空默认假设列表、创建时间和更新时间
- **AND** 会话状态为 `active`
- **AND** API 响应不暴露会话内部自增 ID

#### Scenario: 使用首条用户消息创建会话

- **WHEN** 用户从聊天入口提交第一条商机想法
- **THEN** 系统创建研究需求对齐会话并保存该用户消息
- **AND** 系统生成一条中文助手追问或补充建议
- **AND** 系统不在提交消息时更新结构化研究草稿
- **AND** 系统返回会话状态、已保存消息、当前草稿、缺失字段和可创建状态

#### Scenario: 读取已有对齐会话

- **WHEN** 用户请求一条存在且未软删除的研究需求对齐会话
- **THEN** 系统返回会话公开 UUID、状态、草稿、缺失字段、默认假设、可创建状态、关联研究任务公开 UUID 和消息列表
- **AND** 消息按创建时间正序展示
- **AND** API 响应不暴露会话、消息或研究任务的内部自增 ID

#### Scenario: 不读取不存在或已软删除会话

- **WHEN** 用户请求不存在或已软删除的研究需求对齐会话
- **THEN** 系统返回未找到错误
- **AND** 系统不返回其他会话或消息数据

### Requirement: 系统保存聊天消息并按需更新结构化研究草稿

系统 SHALL 在用户发送消息后保存用户消息并生成自然助手追问，且在用户显式请求更新需求时，将对话内容整理为兼容研究任务创建的结构化草稿。

#### Scenario: 用户消息生成助手追问但不更新草稿

- **WHEN** 用户在对齐会话中发送一条自然语言消息
- **THEN** 系统保存该用户消息
- **AND** 系统调用轻量 LLM 对话或本地 fallback 生成一条中文助手追问或补充建议
- **AND** 系统保存该助手消息
- **AND** 系统可随明确追问型助手消息返回 0-4 个简短候选回复，候选回复不替代用户自由输入
- **AND** 助手消息不使用“已记录”“已更新”“右侧草稿”或“更新需求”等整理状态话术
- **AND** 如果需求信息仍不足，助手消息不能只回复“好的”“收到”“明白”或等价短确认，必须继续提出具体追问
- **AND** 系统不更新研究草稿、缺失字段、默认假设或可创建状态

#### Scenario: 用户短回复后继续追问缺口

- **WHEN** 用户已经提供初始研究方向并继续用候选短回复或短文本补充一个条件
- **THEN** 系统基于已有聊天上下文和最新用户消息判断仍缺少的信息
- **AND** 如果仍缺少预算、渠道、人群、品类、排除条件或供给偏好，系统继续生成中文追问
- **AND** 系统不把短补充句覆盖为新的主研究需求
- **AND** 系统仍不更新研究草稿、缺失字段、默认假设或可创建状态

#### Scenario: 用户更新需求后刷新研究草稿

- **WHEN** 用户在对齐会话中点击更新需求
- **THEN** 系统基于已保存的对话消息和当前草稿生成自然助手回复
- **AND** 系统保存助手消息
- **AND** 系统更新草稿中的自然语言需求、预算、目标渠道、偏好品类、排除品类、目标人群、期望利润、供给来源偏好或其他限制条件
- **AND** 更新后的草稿字段满足现有研究任务创建 payload 的长度和列表格式约束
- **AND** 如果更新需求的 LLM 分析超时或调用失败，系统使用本地 fallback 生成可解析草稿并返回中文助手回复
- **AND** 助手消息不使用“已记录”“已更新”“右侧草稿”或“更新需求”等整理状态话术

#### Scenario: 用户用自然语言要求更新需求

- **WHEN** 用户在对齐会话中发送“信息够了”“先更新需求吧”或等价自然语言意图
- **THEN** 系统保存该用户消息
- **AND** 系统直接执行更新需求流程而不是继续生成追问
- **AND** 系统整理草稿时不把该控制意图文本混入自然语言需求
- **AND** 成功后前端展示更新后的研究草稿和自然助手回复

#### Scenario: 更新需求时后续补充不覆盖原始研究需求

- **WHEN** 会话已经存在自然语言需求且用户继续补充预算、渠道、人群、品类、排除条件或供给偏好后点击更新需求
- **THEN** 系统将补充内容合并到对应结构化字段
- **AND** 系统不把短补充句覆盖成新的自然语言需求
- **AND** 前端在研究草稿面板展示本轮更新后的字段

#### Scenario: 列表字段标准化

- **WHEN** 用户在聊天中用逗号、顿号、换行或自然语言列出多个渠道、品类或供给偏好
- **THEN** 系统将这些内容整理为去空白的字符串列表
- **AND** 系统不保存空字符串列表项
- **AND** 系统保留用户表达中的中文业务含义

#### Scenario: 模型输出无效时保留用户消息

- **WHEN** 对话整理模型返回无法解析、缺少助手回复或草稿字段不符合约束的结果
- **THEN** 系统保留已保存的用户消息
- **AND** 系统优先使用安全 fallback 生成自然追问或可解析草稿更新
- **AND** 系统不创建研究任务
- **AND** 系统不将会话误标记为已准备好创建任务

#### Scenario: 发送追问超时或话术不自然时快速降级

- **WHEN** 用户发送消息且轻量 LLM 追问超时、失败或返回“已记录”“已更新”等整理话术
- **THEN** 系统使用本地 fallback 生成自然追问
- **AND** 系统保存助手消息
- **AND** 系统不向用户展示追问失败错误
- **AND** 系统不更新研究草稿、缺失字段、默认假设或可创建状态

#### Scenario: 本地或测试环境缺少 LLM 凭证

- **WHEN** 系统运行在 local 或 test 环境且未配置 LLM API key
- **THEN** 系统使用确定性 fallback 生成助手回复和草稿更新
- **AND** fallback 结果仍满足对齐会话响应 schema
- **AND** 自动化测试不依赖外部 LLM 服务

### Requirement: 系统判断研究草稿是否足够创建任务

系统 SHALL 根据结构化草稿、缺失字段和默认假设判断用户是否可以确认创建研究任务，并用中文说明仍需补充的信息。

#### Scenario: 草稿缺少核心需求

- **WHEN** 会话草稿缺少非空自然语言需求
- **THEN** 系统将可创建状态标记为不可创建
- **AND** 系统将自然语言需求列入缺失字段或追问重点
- **AND** 助手回复引导用户描述想研究的生意方向

#### Scenario: 草稿达到最小可创建条件

- **WHEN** 会话草稿包含非空自然语言需求
- **AND** 预算已填写或默认假设明确预算未指定
- **AND** 目标渠道已填写或默认假设明确使用中文内容平台
- **AND** 排除品类或限制条件已经被确认，或默认假设明确没有特别排除
- **THEN** 系统将可创建状态标记为可创建
- **AND** 系统返回适合用户确认的中文摘要
- **AND** 系统不要求目标人群、期望利润、供给偏好和偏好品类全部填写后才能创建任务

#### Scenario: 助手一次最多追问两个问题

- **WHEN** 会话草稿仍缺少多个重要条件
- **THEN** 助手回复一次最多提出两个关键澄清问题
- **AND** 系统优先追问会阻断任务创建或明显影响研究方向的条件
- **AND** 系统可以把非阻断字段保留为默认假设或后续可补充信息

### Requirement: 用户可以确认会话并创建研究任务

系统 SHALL 允许用户基于已对齐的会话草稿确认创建研究任务，并启动现有基础商机研究运行。

#### Scenario: 确认可创建会话

- **WHEN** 用户确认一条可创建的研究需求对齐会话
- **THEN** 系统使用当前草稿创建一条研究任务
- **AND** 系统保存自然语言需求、预算、目标渠道、偏好品类、排除品类、目标人群、期望利润、供给来源偏好和其他限制条件
- **AND** 系统启动该研究任务的基础商机研究运行
- **AND** 系统将会话状态更新为 `converted`
- **AND** 系统保存会话关联的研究任务公开 UUID
- **AND** API 返回研究任务公开 UUID 供前端进入任务进度页

#### Scenario: 拒绝确认不可创建会话

- **WHEN** 用户确认一条仍缺少核心需求且不可创建的会话
- **THEN** 系统不创建研究任务
- **AND** 系统返回中文可理解的拒绝原因
- **AND** 响应包含仍需补充的缺失字段
- **AND** 会话保持可继续聊天的状态

#### Scenario: 重复确认已转换会话

- **WHEN** 用户重复确认一条已经转换为研究任务的会话
- **THEN** 系统不创建重复研究任务
- **AND** 系统返回已关联的研究任务公开 UUID
- **AND** 前端可以继续进入该任务进度页

#### Scenario: 启动研究失败但任务已创建

- **WHEN** 用户确认会话后研究任务创建成功但启动运行失败
- **THEN** 系统返回中文可理解的启动失败信息
- **AND** 系统返回已创建研究任务的公开 UUID
- **AND** 前端可以引导用户进入任务进度页查看状态或重新运行

### Requirement: 聊天阶段保持需求对齐边界

系统 SHALL 将研究需求对齐会话限定为前置需求澄清能力，不在聊天阶段生成商机结果或宣称完成市场调研。

#### Scenario: 聊天阶段不生成商机结果

- **WHEN** 用户在研究需求对齐会话中发送消息
- **THEN** 系统不创建商机推荐
- **AND** 系统不生成商机排行榜、商机详情、基础报告、来源列表、RAG evidence chunks、竞品参考、验证预算、风险复核或行动计划
- **AND** 系统不投递基础商机研究 Celery 任务

#### Scenario: 聊天阶段不执行外部前置调研

- **WHEN** 系统整理聊天需求草稿
- **THEN** 系统不调用 Tavily、Playwright、网页正文提取、向量检索或 RAG 流程来补全草稿
- **AND** 助手回复不得宣称已经完成公开市场、供给、竞品、风险或来源核验
- **AND** 助手回复将草稿表达为待确认研究需求或默认假设

### Requirement: 系统提供对话整理可观测性和安全错误摘要

系统 SHALL 为研究需求对齐会话的消息处理和 LLM 整理过程记录可排障信息，同时避免向用户暴露敏感细节。

#### Scenario: 记录对话整理运行信息

- **WHEN** 系统处理一条用户聊天消息
- **THEN** 系统记录会话公开 UUID、消息公开 UUID、处理阶段、运行状态和耗时
- **AND** 如果 tracing 已启用，系统记录可关联的 trace metadata
- **AND** 应用日志可以保留错误类型和阶段名用于开发者排障

#### Scenario: 返回安全错误摘要

- **WHEN** 对话整理或确认创建过程失败
- **THEN** 系统向用户返回中文可理解的安全错误摘要
- **AND** 响应不暴露完整 prompt、模型原始输出、API key、内部自增 ID、原始异常堆栈或敏感配置
- **AND** 失败不会修改已创建研究任务的结果状态或已有商机结果

### Requirement: 前端展示聊天式需求对齐体验

前端 SHALL 提供聊天式研究需求对齐体验，让用户在确认前看到对话、草稿、缺失条件、默认假设和启动入口。

#### Scenario: 查看聊天会话页面

- **WHEN** 用户打开一条研究需求对齐会话
- **THEN** 页面展示中文聊天消息列表、消息输入框、研究草稿摘要、缺失条件和默认假设
- **AND** 如果同一浏览器会话中存在未确认的对齐会话，页面恢复该会话消息和草稿状态
- **AND** 聊天消息列表在桌面端适配浏览器可用高度，消息增多时仅在消息区域内部滚动且不撑高整页主体
- **AND** 新消息或思考中状态出现后，聊天消息列表自动滚动到最新位置
- **AND** 研究草稿面板与聊天卡在桌面端等高适配视口，字段列表增多时仅在面板内容区内部滚动且不撑高整页主体
- **AND** 聊天消息在气泡旁展示对应发送者头像，用户使用用户头像，商机顾问使用 MarketPilot 头像
- **AND** 聊天气泡内不再展示“你”或“商机顾问”作为可见身份标签
- **AND** 页面展示该会话当前是否可以创建研究任务
- **AND** 页面不展示静态商机结果或静态报告作为聊天结果

#### Scenario: 切换聊天和表单模式

- **WHEN** 用户在同一新建研究工作台中切换聊天模式和表单模式
- **THEN** 前端保留切换前聊天模式中的消息、输入框内容和研究草稿状态
- **AND** 前端保留切换前表单模式中已经填写的字段内容
- **AND** 前端不通过不同路由或重新创建页面来丢弃任一模式的未提交内容

#### Scenario: 发送聊天消息

- **WHEN** 用户在聊天页面发送消息
- **THEN** 前端立即在聊天消息列表中展示该用户消息
- **AND** 前端在聊天对话中展示商机顾问头像和“思考中”loading 气泡
- **AND** 发送按钮不展示发送中的 loading 文案或 spinner
- **AND** 成功后展示用户消息和助手追问
- **AND** 助手追问以自然聊天方式补充问题，不展示“已记录”“已更新”类整理状态
- **AND** 前端不自动更新研究草稿
- **AND** 失败时展示中文可理解错误，并允许用户继续编辑或重试

#### Scenario: 点击候选短回复

- **WHEN** 商机顾问明确追问后存在可推断的简短用户回复选项
- **THEN** 前端在最新一条商机顾问回复下方展示 API 返回的候选短回复
- **AND** 用户点击任一候选短回复时，前端将该短回复作为用户消息发送
- **AND** 用户仍可忽略候选短回复并在编辑框中自由输入
- **AND** 前端不在旧的商机顾问消息下持续展示过期候选短回复
- **AND** 如果最新助手消息只是说明信息已足够或提示可继续补充，前端不展示候选短回复

#### Scenario: 带入表单编辑聊天草稿

- **WHEN** 用户已经通过更新需求生成结构化研究草稿
- **THEN** 前端提供将该草稿带入表单模式编辑的入口
- **AND** 用户点击后切换到表单模式并填入草稿字段
- **AND** 用户可以在表单中继续修改并使用既有创建并启动流程

#### Scenario: 手动更新需求

- **WHEN** 用户点击更新需求
- **THEN** 前端在更新需求按钮上展示 loading 动效
- **AND** 成功后展示自然助手回复和更新后的研究草稿
- **AND** 聊天消息不展示结构化字段 badge
- **AND** 失败时展示中文可理解错误，并允许用户继续补充或重试

#### Scenario: 编辑框 Enter 快捷发送

- **WHEN** 用户在聊天编辑框中按下 Enter
- **THEN** 前端发送当前消息
- **AND** 用户使用 Shift+Enter 时保留换行输入
- **AND** 中文输入法组合输入过程中按下 Enter 不触发发送

#### Scenario: 确认并启动研究

- **WHEN** 会话可创建且用户点击确认启动
- **THEN** 前端调用会话确认 API
- **AND** 成功后跳转到新研究任务的进度页
- **AND** 前端不要求用户再手动复制草稿到表单页面

#### Scenario: 小屏查看聊天体验

- **WHEN** 用户在手机或窄屏视口打开聊天页面
- **THEN** 页面优先展示聊天消息、输入框和确认启动操作
- **AND** 研究草稿可以在同页下方、折叠区或切换区查看
- **AND** 页面不出现需要横向滚动才能理解主操作的布局
