## 背景

当前 MarketPilot 的新建研究入口以表单为主，字段已经和 `ResearchTaskCreate` 对齐：自然语言需求、预算、目标渠道、偏好品类、排除品类、目标人群、期望利润、供给来源偏好和其他限制条件。用户如果已经想清楚这些条件，表单效率高；但如果用户只有模糊方向，例如“想做小红书上适合新手的小生意”，表单会把需求澄清压力提前交给用户。

本 change 新增聊天式研究需求对齐能力。它位于正式研究任务之前，只负责把模糊输入整理成结构化研究草稿，并在用户确认后复用现有研究任务创建和启动流程。

## 目标 / 非目标

**目标：**

- 提供用户侧新建研究聊天模式，让用户通过聊天完成研究需求对齐。
- 保存研究需求对齐会话、消息、结构化草稿、缺失字段、默认假设和可创建状态。
- 用户发送消息后生成自然中文追问或补充建议，但不刷新结构化草稿，也不展示“已记录/已更新”类整理话术；点击“更新需求”后才更新可用于创建研究任务的结构化草稿。
- 支持用户在草稿足够完整后确认创建并启动研究任务，然后进入任务进度页。
- 对聊天阶段的 LLM 调用提供可排障日志或 trace metadata，失败时返回中文可理解错误。
- 保持现有研究任务、商机生成、来源收集、RAG、评测和报告分享流程不变。

**非目标：**

- 不实现报告完成后的追问问答或基于已生成报告的客服式问答。
- 不在聊天阶段执行外部搜索、网页抓取、RAG 检索、竞品深挖或多 Agent 调研。
- 不让聊天阶段直接生成商机结果、商机排行榜、报告、来源结论或验证计划。
- 不替代表单新建研究入口；表单模式继续可用。
- 不新增认证、配额、多人会话或跨设备同步能力。

## 技术决策

### 决策 1：新增独立 `research-intake-conversations` 能力

聊天需求对齐使用独立后端模块和前端页面，不把对话消息塞进 `research_tasks` 表，也不把研究任务状态扩展为“聊天中”。会话在用户确认前是草稿上下文；确认后才创建正式研究任务。

备选方案是直接让每条聊天草稿成为 `created` 状态研究任务，但这会污染“我的研究”列表，并且用户还没确认时就产生正式业务记录。独立会话可以把草稿和正式任务边界分清楚。

### 决策 2：会话和消息使用持久化表

新增两类表：

- `research_intake_conversations`：保存公开 UUID、状态、草稿 payload、缺失字段、默认假设、可创建状态、失败摘要和可选的关联研究任务。
- `research_intake_messages`：保存公开 UUID、会话外键、角色、消息正文、结构化增量或 metadata。

两张表都遵循项目数据库通用字段规范，包含数据库内部自增 `id`、`created_at`、`updated_at` 和 `deleted_at`。对外 API 只暴露公开 UUID，不暴露内部自增 ID。默认读取只返回 `deleted_at` 为空的会话和消息；软删除会话后，默认不再读取其消息。

### 决策 3：发送后追问，显式更新草稿

用户发送消息时，前端先在聊天流中展示用户消息和商机顾问“思考中”loading，API 同步保存用户消息，并通过轻量 LLM 对话生成一条自然中文追问；这一步不要求固定 JSON 输出，不应用结构化草稿增量、不刷新缺失字段、不改变可创建状态。若轻量追问超时、失败或回复包含“已记录/已更新”等整理话术，服务层使用本地 fallback 快速生成自然追问，避免发送按钮长时间 loading。用户点击“更新需求”时，API 再同步完成：

1. 基于最近消息和当前草稿调用 LLM 整理服务。
2. 保存助手消息和草稿更新。
3. 返回会话摘要、最近消息、草稿、缺失字段和可创建状态。

备选方案是把每轮对话投递 Celery，但聊天体验需要低延迟反馈，且本阶段不做外部检索或长耗时分析。同步 API 更小、更符合“客服式澄清”的交互。

服务层会随明确追问型助手消息返回 0-4 个可点选短回复；前端只在最新一条商机顾问追问下方展示这些候选词。点击候选词等价于发送一条普通用户消息。候选词只是降低输入成本，不替代用户自由输入，也不影响用户手动输入。若助手消息只是说明信息已足够或提示可继续补充，服务层不返回候选词。

发送阶段的服务层会用最近用户消息构造临时草稿来决定下一轮追问，但不会把这个临时草稿写入会话草稿字段。这样用户点击“先拿样品”这类短回复后，系统仍能结合前文继续追问预算、渠道或排除条件；如果 LLM 只返回“好的/收到/明白”这类短确认，服务层会判定为不可用并切换到本地追问 fallback。

### 决策 4：更新需求使用固定 JSON 契约

更新需求整理服务要求模型返回固定结构：

- `assistant_message`：中文助手回复，最多追问两个关键问题。
- `draft_update`：可覆盖到研究任务创建 payload 的字段增量。
- `missing_fields`：仍需确认的关键字段。
- `assumptions`：系统采用的默认假设。
- `readiness_status`：`needs_clarification` 或 `ready`。
- `can_create_task`：是否允许用户确认创建任务。

服务层负责校验字段长度、列表标准化和空值清理，最终草稿必须兼容现有 `ResearchTaskCreate` schema。模型输出无效、调用失败或超时时，系统使用安全 fallback 生成可解析结果，保留用户消息，并避免向用户暴露模型原始错误。

### 决策 5：就绪判断采用“最小可执行”而不是字段全填满

会话不要求用户填写所有字段。最小可创建条件为：

- 有非空 `brief`。
- 预算已填写，或默认假设中明确预算未指定。
- 目标渠道已填写，或默认假设中明确使用中文内容平台。
- 排除品类或限制条件至少被询问过；用户可以回答“没有特别排除”。

目标人群、期望利润、供给偏好和偏好品类用于提升研究质量，但不阻断用户创建任务。这样避免聊天入口变成冗长问卷。

### 决策 6：确认启动复用现有研究任务 API 和进度页

当用户点击确认启动时，前端调用新的会话确认 API；后端用当前草稿构造 `ResearchTaskCreate`，创建研究任务，并调用现有启动研究运行逻辑。成功后返回研究任务公开 UUID，前端跳转到 `/research/tasks/{task_uuid}`。

如果启动失败但任务已创建，API 返回可理解错误和任务 UUID，前端允许用户进入任务进度页继续查看或重新运行。会话记录关联的研究任务 UUID，避免重复确认创建多条任务。

### 决策 7：产品入口保持“新建研究”主导航不扩张

全局导航仍保持“新建研究 / 我的研究”。`/` 是统一的新建研究工作台，聊天模式为默认体验，表单模式通过工作区顶部面包屑行右侧按钮切换；`/research/new` 和 `/research/chat` 作为旧入口重定向回 `/`，避免路径和面包屑把两种模式表现成不同页面。

聊天模式和表单模式在同一工作台内保持挂载，只切换可见性。这样用户在聊天中已经输入的消息、待发送内容、右侧研究草稿，以及表单里已经填写的字段，不会因为点右上角模式切换按钮而丢失。

前端会在同一浏览器会话中保存当前未确认对齐会话的公开 UUID，并在用户刷新或回到首页工作台时尝试恢复该会话。已确认并创建研究任务的会话不作为默认聊天上下文恢复，避免用户回到首页后继续停留在旧任务草稿上。

聊天草稿生成后，用户可以把草稿带入表单模式继续编辑。该入口只复用当前结构化草稿填充表单字段，不创建任务、不修改已保存会话消息，也不丢弃聊天状态。

### 决策 8：记录可排障信息但不暴露敏感细节

聊天整理服务记录会话 UUID、消息 UUID、调用阶段、耗时、结果状态和安全错误摘要。若接入 LangSmith tracing，可记录 trace id 或 trace URL 到会话 metadata 或应用日志；用户侧响应不得暴露完整 prompt、模型原始输出、内部自增 ID、API key、原始异常堆栈或敏感配置。

## 风险 / 取舍

- 聊天入口可能让用户误以为已经完成市场调研 → 页面和助手文案明确表达“正在整理研究需求”，不宣称已完成公开来源、供给、竞品或风险核验。
- LLM 提取字段可能误解用户意图 → 草稿面板持续展示结构化结果，用户确认前可以继续补充，也可以带入表单直接编辑，确认时以草稿或表单提交内容为准。
- 同步 LLM 调用可能变慢或失败 → 发送阶段使用短超时和本地 fallback，更新需求阶段也设置超时并回退到本地解析；前端在聊天流中展示“思考中”loading，并只在“更新需求”按钮上展示按钮 loading；失败时保留用户消息并允许继续补充或重试，不影响已有表单新建研究。
- 会话草稿和正式任务可能重复创建 → 会话确认后记录关联研究任务；重复确认返回已有任务或拒绝重复创建。
- 新增持久化表增加迁移复杂度 → 表结构保持简单，只保存会话、消息和 JSON 草稿，不引入复杂状态机。
- fallback 可能无法像真实模型一样自然追问 → 本地和测试环境优先保证可预测、可验证；线上发送阶段若 LLM 慢或话术不合适也优先返回自然 fallback，保证聊天连续性。

## 迁移计划

- 新增 Alembic 迁移创建会话和消息表，不修改现有研究任务、商机、来源、RAG 或质量评测表。
- 新增 API 和前端页面后，现有表单创建研究能力保持可用，但入口收束到首页工作台的表单模式。
- 部署时如果未配置 LLM key，本地或测试环境使用确定性 fallback；线上环境返回中文可理解错误并保留消息。
- 回滚时可下线聊天模式和相关 API；已创建的正式研究任务不受影响。

## 开放问题

- 是否需要限制单个会话最大消息数或过期时间，本 change 先通过后端常量控制，后续若有真实用户增长再单独治理。
- 是否需要把聊天整理 trace 链接展示给内部质量面板，本 change 只要求记录可排障 metadata，不要求新增内部查看入口。
