## 背景

MarketPilot 当前已经跑通从新建研究、执行任务、查看商机推荐、阅读基础报告到分享报告的闭环，但一级导航仍保留早期骨架里的研究任务、商机推荐、最终报告和研究历史并列入口。

实际数据模型里，商机推荐、基础报告、来源和行动计划都依附于同一条研究任务；研究历史也只是同一批研究任务的另一种列表视图。继续并列展示会让用户误以为这些是不同业务对象，尤其在完成任务后从任务列表进入“结果”、从历史列表进入“报告”时，路径语义不一致。

本 change 不改变数据库、后端 API 或 Agent 流程，只整理前端信息架构、路由入口和列表主操作。

## 目标 / 非目标

**目标：**

- 将一级导航收束为“新建研究”和“我的研究”这类不依赖任务上下文的入口。
- 用统一研究列表承载原研究任务和研究历史的查看、筛选、继续、重新运行能力。
- 让已完成任务只有一个清晰主操作：查看研究结果。
- 让商机推荐、基础报告、来源等结果页面保持任务内导航，不再作为互相竞争的全局入口。
- 保留 `/history`、`/opportunities` 和 `/reports` 这类既有路径的可理解空状态或兼容跳转，避免用户访问旧路径时迷路。

**非目标：**

- 不新增数据库表、字段或迁移。
- 不修改研究任务 API、商机结果 API、报告分享 API 或 readiness API 的响应契约。
- 不改变 Agent、RAG、来源收集、商机排序、报告生成或分享快照逻辑。
- 不做跨任务商机库、跨任务报告库、搜索或分页能力。

## 技术决策

1. 一级导航合并为“我的研究”，而不是保留“研究任务”和“研究历史”两个入口。

   原因：两者当前读取同一个 `research-tasks` 数据源，用户可见差别只是字段和按钮文案。合并后可以用状态筛选表达“进行中 / 已完成 / 失败 / 全部”，避免两张相似列表争夺注意力。

   备选方案：保留研究历史并改名为“报告档案”。这个方案适合未来有稳定报告归档、搜索和分享管理能力时再引入；当前会继续制造重复入口。

2. 已完成任务的默认主操作进入基础报告页，按钮文案统一为“查看研究结果”。

   原因：基础报告是当前最完整的单条研究摘要，已经能承载商机排序、增强分析、来源、行动计划和分享入口。商机推荐仍通过任务内导航保留，不需要作为完成任务的另一套默认入口。

   备选方案：默认进入商机推荐页。这个方案保留现有任务列表行为，但无法解释为什么历史列表点的是报告，也会弱化报告作为最终交付物的地位。

3. `/history` 保留为兼容入口，但不再展示独立历史列表。

   原因：直接删除路径会让旧链接或测试入口失效；兼容跳转或渲染统一研究列表能减少迁移风险。实现时可以选择服务端 redirect 到 `/research/tasks?status=completed`，也可以复用统一列表组件并默认筛选已完成。

   备选方案：完全移除 `/history` 页面。这样更干净，但对现有导航测试和用户手动访问不够友好。

4. `/opportunities` 和 `/reports` 的无任务上下文入口只做选择引导。

   原因：结果页必须绑定任务 UUID，否则容易把静态演示或其他任务结果误认为当前结果。保留可行动中文空状态或任务选择入口，比把它们放在一级导航更清楚。

   备选方案：从这些入口自动选择最近完成任务。这个方案速度快，但用户可能不知道自己正在看哪一次研究，演示时也容易误读。

## 迁移计划

- 前端先调整导航配置、页面标题和 i18n 文案，不涉及后端部署顺序。
- 统一研究列表继续使用现有 `fetchResearchTasks`，默认只读取未软删除任务。
- `/history` 作为兼容入口接入统一研究列表或跳转到统一列表的已完成筛选。
- 完成任务主操作统一指向基础报告页；报告页继续提供任务内商机推荐入口。
- 回滚时可恢复旧导航配置和历史页面组件，不需要数据库回滚。

## 风险 / 取舍

- [Risk] 用户习惯从一级导航直接点击“商机推荐”或“最终报告”。→ Mitigation: 无上下文结果入口保留可行动空状态，引导用户从“我的研究”选择任务。
- [Risk] “我的研究”同时承载运行中和历史记录，列表信息变多。→ Mitigation: 使用状态筛选或分组，并让每条记录只有一个状态相关主操作。
- [Risk] 默认进入报告可能让只想快速比较商机的用户多一次点击。→ Mitigation: 报告页保留显眼的任务内“商机推荐”入口，且报告本身展示商机排序摘要。
- [Risk] 旧 `/history` 语义和新“我的研究”默认筛选不一致。→ Mitigation: 兼容入口默认筛选已完成或清楚标注这是统一研究记录的一部分。

## 开放问题

- 统一列表的默认筛选使用“全部”还是“进行中优先”？建议实现时先用“全部”，并按状态与创建时间让需要处理的任务靠前。
- 一级导航文案最终使用“我的研究”还是“研究记录”？建议先用“我的研究”，因为它同时覆盖待启动、运行中、失败和已完成记录。
